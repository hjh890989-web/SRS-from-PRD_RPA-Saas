## :white_check_mark: Task Breakdown (실행 계획)

### Phase A. 프로젝트 초기 Prisma 설치 및 설정

- [ ] **A-1. Prisma 설치**: `npm install prisma @prisma/client` (devDependencies / dependencies 구분)
- [ ] **A-2. Prisma 초기화**: `npx prisma init` 실행 → `prisma/schema.prisma` 및 `.env` 파일 생성 확인
- [ ] **A-3. `.gitignore` 검토**: `.env`, `.env.local`, `.env.cloud` 등 환경 변수 파일 Git 제외 처리 확인

### Phase B. 이중 DB Provider 구성 (SQLite ↔ PostgreSQL 전환 구조)

- [ ] **B-1. `schema.prisma` provider 설정**: `env("DATABASE_PROVIDER")` 환경 변수로 `sqlite` / `postgresql` 동적 전환 구조 설계
  ```prisma
  // prisma/schema.prisma
  datasource db {
    provider = env("DATABASE_PROVIDER") // "sqlite" | "postgresql"
    url      = env("DATABASE_URL")
  }

  generator client {
    provider = "prisma-client-js"
  }
  ```
- [ ] **B-2. 로컬(dev) 환경변수 설정** (`.env.local`):
  ```env
  DATABASE_PROVIDER="sqlite"
  DATABASE_URL="file:./dev.db"
  ```
- [ ] **B-3. MVP(cloud) 환경변수 설정** (`.env.cloud`):
  ```env
  DATABASE_PROVIDER="postgresql"
  DATABASE_URL="postgresql://[user]:[password]@[host]:5432/[dbname]?schema=public&pgbouncer=true&connect_timeout=15"
  DIRECT_URL="postgresql://[user]:[password]@[host]:5432/[dbname]?schema=public"
  ```
  > :warning: Supabase + PgBouncer(연결 풀링) 환경에서는 마이그레이션용 `DIRECT_URL`을 반드시 별도 설정해야 함 (Prisma 공식 권고사항)
- [ ] **B-4. `schema.prisma`에 `directUrl` 추가**:
  ```prisma
  datasource db {
    provider  = env("DATABASE_PROVIDER")
    url       = env("DATABASE_URL")
    directUrl = env("DIRECT_URL") // Supabase PgBouncer 마이그레이션 전용
  }
  ```

### Phase C. Prisma Client 싱글톤 인스턴스 구현

- [ ] **C-1. `lib/prisma.ts` 싱글톤 파일 생성**: Next.js 개발 서버의 Hot Reload로 인한 PrismaClient 인스턴스 중복 생성 방지
  ```typescript
  // lib/prisma.ts
  import { PrismaClient } from '@prisma/client';

  const globalForPrisma = globalThis as unknown as {
    prisma: PrismaClient | undefined;
  };

  export const prisma =
    globalForPrisma.prisma ??
    new PrismaClient({
      log: process.env.NODE_ENV === 'development' ? ['query', 'error', 'warn'] : ['error'],
    });

  if (process.env.NODE_ENV !== 'production') globalForPrisma.prisma = prisma;
  ```

### Phase D. 첫 마이그레이션 실행 및 연결 검증

- [ ] **D-1. SQLite 초기 마이그레이션 실행** (개발 환경 검증용):
  ```bash
  # .env.local 환경으로 첫 마이그레이션
  ```

- [ ] **D-2. Supabase 프로젝트 생성 및 연결 정보 확보**: Supabase 대시보드에서 `Project URL`, `API Key (anon/service_role)`, `Database URL` 확인
- [ ] **D-3. Supabase PostgreSQL 연결 검증**:

- [ ] **D-4. `npx prisma studio` 실행**: 양 환경에서 Prisma Studio로 DB 연결 정상 여부 시각적 확인

## :test_tube: Acceptance Criteria (BDD/GWT)

### Scenario 1: 로컬 개발 환경(SQLite) DB 연결 성공

- **Given**: `.env.local`에 `DATABASE_PROVIDER="sqlite"` 및 `DATABASE_URL="file:./dev.db"` 설정이 완료되어 있음
- **When**: `npm run db:migrate:dev` 명령을 실행함
- **Then**: `prisma/migrations/` 디렉토리에 마이그레이션 파일이 생성되고, 로컬에 `dev.db` SQLite 파일이 생성되며, `npx prisma studio` 실행 시 빈 스키마가 정상적으로 표시된다.

### Scenario 2: MVP 클라우드(Supabase PostgreSQL) DB 연결 성공

- **Given**: `.env.cloud`에 Supabase `DATABASE_URL` 및 `DIRECT_URL`이 올바르게 설정되어 있음
- **When**: `npm run db:migrate:prod` 명령을 실행함 (또는 초기 검증 시 `db push`)
- **Then**: Supabase 대시보드 Table Editor에 마이그레이션이 적용되고, Prisma Client를 통한 쿼리가 오류 없이 실행된다.

### Scenario 3: 환경 전환 시 스키마 일관성 보장

- **Given**: `schema.prisma` 파일에서 `provider = env("DATABASE_PROVIDER")` 구조가 적용되어 있음
- **When**: `DATABASE_PROVIDER` 환경 변수를 `sqlite`에서 `postgresql`로 (또는 역방향으로) 변경하고 `prisma generate`를 재실행함
- **Then**: 동일한 `schema.prisma` 정의로 양 환경 모두에서 Prisma Client가 정상 생성되고, 타입 오류가 없다.

### Scenario 4: PrismaClient 중복 인스턴스 방지 (개발 환경 Hot Reload)

- **Given**: Next.js 개발 서버(`npm run dev`)가 실행 중이며, `lib/prisma.ts` 싱글톤 패턴이 적용되어 있음
- **When**: 소스 코드를 수정하여 Hot Reload가 발생함
- **Then**: 콘솔에 `warn(prisma-client) There are already 10 instances...` 경고 메시지가 출력되지 않으며, 단일 PrismaClient 인스턴스만 유지된다.

### Scenario 5: 잘못된 DB 연결 정보 입력 시 명확한 오류 출력

- **Given**: `.env.cloud`에 잘못된 Supabase `DATABASE_URL`이 입력되어 있음
- **When**: `npx prisma db push` 또는 서버 구동 시 DB 쿼리가 실행됨
- **Then**: 서버가 크래시되지 않고, 표준 Prisma 연결 오류 메시지(`PrismaClientInitializationError`)가 출력되어 개발자가 원인을 식별할 수 있다.

## :memo: Implementation Notes (개발자 참고)

### Supabase 연결 시 PgBouncer 주의사항

Supabase는 기본적으로 연결 풀러(PgBouncer)를 사용합니다. Prisma Migrate 실행 시에는 PgBouncer를 우회하는 **Direct Connection URL**을 반드시 사용해야 합니다.

- `DATABASE_URL`: Transaction Mode URL (포트 `6543`) — 일반 쿼리용
- `DIRECT_URL`: Session Mode URL (포트 `5432`) — 마이그레이션 전용

```env
# .env.cloud 예시
DATABASE_URL="postgresql://postgres.[ref]:[password]@aws-0-[region].pooler.supabase.com:6543/postgres?pgbouncer=true"
DIRECT_URL="postgresql://postgres.[ref]:[password]@aws-0-[region].pooler.supabase.com:5432/postgres"
```

### SQLite → PostgreSQL 마이그레이션 파일 호환성

SQLite와 PostgreSQL은 일부 데이터 타입 구문이 다릅니다 (예: `AUTOINCREMENT` vs `SERIAL`). Prisma는 이를 자동으로 처리하지만, 직접 SQL 마이그레이션 파일을 수정할 경우 반드시 양 환경에서 테스트해야 합니다.

### Prisma Studio 포트 충돌 방지

`npx prisma studio`는 기본적으로 `5555` 포트를 사용합니다. Next.js 개발 서버(`3000`)와 충돌하지 않으나, 다른 서비스와 충돌 시 `--port` 옵션으로 포트를 변경하세요.
