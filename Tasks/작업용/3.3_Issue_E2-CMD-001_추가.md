## :dart: Summary
- **기능명**: [E2-CMD-001] Lot 시간순 병합 로직 구현 (타임스탬프 중복/역전 검증 포함, 오병합 0건)
- **목적**: 감사 리포트 생성의 핵심 전처리 단계로, 여러 LOG_ENTRY 데이터를 LOT 단위로 시간순 병합한다. 타임스탬프 중복·역전이 존재하는 경우 자동 병합을 차단하고 충돌 Lot 목록을 표시하여 수동 확인을 요청함으로써, **오병합 0건**과 **Lot 정확도 ≥99%**를 보장한다. 이 모듈은 E2 원클릭 감사 리포트의 데이터 정합성 기반을 제공한다.

## :white_check_mark: Task Breakdown (실행 계획)
- [ ] **1. 도메인 서비스 생성**: `lib/services/lot-merge-service.ts` — Lot 병합 핵심 비즈니스 로직
- [ ] **2. LOG_ENTRY → LOT 매핑 로직**:
  - 지정된 `lot_ids` 또는 `date_range`에 해당하는 LOG_ENTRY 조회
  - LOG_ENTRY의 `captured_at` 기준으로 시간순 정렬
  - 동일 LOT에 속하는 LOG_ENTRY 그룹핑 (FK: `work_order_id` 기반)
- [ ] **3. 타임스탬프 무결성 검증 엔진**: `lib/services/timestamp-validator.ts`
  - **중복 감지**: 동일 LOT 내 `captured_at` 타임스탬프가 밀리초 단위까지 동일한 항목 탐지
  - **역전 감지**: LOT 내 LOG_ENTRY의 `captured_at` 순서가 `work_order` 진행 순서와 불일치하는 경우 탐지
  - **갭 분석**: LOT 시작~종료 구간 내 비정상적 시간 공백(>30분) 감지
  - 각 검증 결과를 `ConflictReport` 타입으로 구조화 반환
- [ ] **4. 충돌 처리 분기 로직**:
  - 충돌 없음 → `MergeResult(status: "SUCCESS", merged_data: [...])` 반환
  - 충돌 존재 → `MergeResult(status: "CONFLICT", conflicts: [...], blocked: true)` 반환
  - 충돌 시 **자동 병합 차단** + 충돌 LOT 목록 + 충돌 유형(DUPLICATE/REVERSAL/GAP) 반환
- [ ] **5. 수동 병합 순서 지정 API**: `lib/services/lot-merge-service.ts#resolveConflict()`
  - 관리자가 충돌 LOT에 대해 수동으로 순서를 지정하는 로직
  - 수동 재정렬 후 재검증 → 통과 시 병합 진행
- [ ] **6. 병합 결과 객체 (Merged Lot Data)**: 
  - 각 LOT별 시간순 정렬된 LOG_ENTRY 배열
  - LOT 메타데이터 (lot_number, start_time, end_time, entry_count)
  - 병합 무결성 해시 (데이터 변조 방지)
- [ ] **7. 감사 로그 기록**: 병합 시도, 충돌 감지, 수동 해결 전 과정 감사 로그 자동 기록

## :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: 정상적인 Lot 시간순 병합**
- **Given**: 3개 LOT (LOT-001, LOT-002, LOT-003)에 각각 10건의 LOG_ENTRY가 존재하며, 모든 타임스탬프가 정상 시간순이다
- **When**: `POST /api/v1/audit-reports`에서 Lot 병합 로직이 호출된다
- **Then**: 30건의 LOG_ENTRY가 LOT별 시간순으로 정확하게 병합된다. `MergeResult(status: "SUCCESS")`가 반환되고, Lot 정확도 ≥99%가 달성된다. 각 LOT의 `start_time`~`end_time` 범위가 정확하다.

**Scenario 2: 타임스탬프 중복 감지 → 자동 병합 차단**
- **Given**: LOT-001 내에 2건의 LOG_ENTRY가 동일한 `captured_at` 값(2026-04-19T10:30:00.000Z)을 가진다
- **When**: Lot 병합이 시도된다
- **Then**: 타임스탬프 중복이 감지되어 `MergeResult(status: "CONFLICT", conflicts: [{ type: "DUPLICATE", lot_number: "LOT-001", entries: [...] }], blocked: true)` 반환. **자동 병합 차단**. 충돌 Lot 목록이 ≤5초 내 표시된다. **오병합 0건**.

**Scenario 3: 타임스탬프 역전 감지 → 자동 병합 차단**
- **Given**: LOT-002 내에서 LOG_ENTRY #5의 `captured_at`이 LOG_ENTRY #4보다 이전 시간이다 (역전)
- **When**: Lot 병합이 시도된다
- **Then**: 타임스탬프 역전이 감지되어 `MergeResult(status: "CONFLICT", conflicts: [{ type: "REVERSAL", lot_number: "LOT-002", entries: [...] }], blocked: true)` 반환. 수동 확인 요청 트리거. **경고 ≤5초**.

**Scenario 4: 충돌 수동 해결 후 재병합**
- **Given**: Scenario 2/3에서 감지된 충돌에 대해 관리자가 수동으로 올바른 순서를 지정한다
- **When**: `resolveConflict(lotId, manualOrder)` 호출 후 재병합을 시도한다
- **Then**: 재검증 통과 → `MergeResult(status: "SUCCESS")` 반환. 수동 해결 과정이 감사 로그에 기록된다.

**Scenario 5: 대규모 LOT 병합 (100 Lot) 성능**
- **Given**: 100개 LOT, 각 LOT당 50건의 LOG_ENTRY (총 5,000건)가 존재한다
- **When**: 전체 Lot 병합이 요청된다
- **Then**: 병합이 완료되어 결과가 반환된다. LOG_ENTRY 정렬 + 검증 + 그룹핑 전 과정이 합리적 시간(≤10초) 내 완료된다.

**Scenario 6: 승인된(APPROVED) LOG_ENTRY만 병합 대상**
- **Given**: LOT-001에 10건 LOG_ENTRY 중 3건이 `status=PENDING`, 1건이 `status=REJECTED`이다
- **When**: Lot 병합이 호출된다
- **Then**: `status=APPROVED`인 6건만 병합 대상으로 포함된다. PENDING/REJECTED 건은 제외된다. 미승인 데이터가 감사 리포트에 포함되는 것을 방지한다.

## :gear: Technical & Non-Functional Constraints
- **정확도**: Lot 병합 정확도 ≥99% (REQ-FUNC-009)
- **오병합 방지**: 타임스탬프 충돌 시 자동 병합 100% 차단, 오병합 0건 (REQ-FUNC-013)
- **충돌 경고**: 충돌 감지 → 목록 표시 ≤5초 (REQ-FUNC-013)
- **성능**: 100 LOT(5,000 LOG_ENTRY) 병합 ≤10초 (Vercel 서버리스 환경)
- **데이터 필터링**: `status=APPROVED`인 LOG_ENTRY만 병합 대상 (HITL ADR-3)
- **무결성**: 병합 결과에 무결성 해시 포함 (데이터 변조 감지)
- **감사 추적**: 병합 시도/충돌/해결 전 과정 감사 로그 기록 (REQ-NF-023)
- **서버 제한**: Vercel Serverless ≤10초. 대규모 병합 시 분할 처리 또는 클라이언트 사이드 처리 고려 (CON-07)
