## :dart: Summary
- **기능명**: [E1-CMD-002] 모바일 카메라 촬영 → Gemini Vision API 이미지 파싱 + LOG_ENTRY 저장 (status=PENDING)
- **목적**: 작업자가 모바일 카메라로 촬영한 완성품/계기판 이미지를 Gemini Vision API로 파싱하여 상태 값(온도, 압력, 수량 등)을 구조화된 데이터로 추출하고, LOG_ENTRY 테이블에 `status=PENDING`으로 저장한다. 수기 기록을 대체하여 현장 데이터 수집 속도와 정확도를 혁신적으로 향상시키는 핵심 기능이다.

- [ ] **5. 파싱 실패 감지 로직**:

  - 실패 시 LOG_ENTRY에 저장하지 않음 (오 데이터 기록 0건)

  - `parsed_data`: Vision 파싱 결과 (구조화된 JSON)
  - `status`: `PENDING` (기본값)
  - `captured_at`: 서버 수신 타임스탬프
- [ ] **7. 에러 핸들링 + 재촬영 알림 연동 준비**:

  - Gemini API 타임아웃(8초) → 재시도 1회 → 실패 시 에러 응답

## :test_tube: Acceptance Criteria (BDD/GWT)

**Scenario 1: 정상적인 이미지 촬영 → 상태 값 파싱 + 로깅**
- **Given**: 작업자(OPERATOR 역할)가 유효한 계기판 이미지(JPEG, 3MB, 1920x1080)와 유효한 `factory_id`, `line_id`를 보유한다
- **When**: 모바일 카메라로 촬영하여 `POST /api/v1/log-entries`로 이미지를 전송한다
- **Then**: Gemini Vision API가 호출되어 상태 값(온도, 압력 등)이 JSON으로 추출된다. LOG_ENTRY 테이블에 `source_type=VISION`, `status=PENDING`, `parsed_data`에 구조화된 파싱 결과가 저장된다. 201 Created 반환. 파싱 성공률 ≥85%, 처리 ≤5초.

**Scenario 2: 렌즈 오염·역광으로 인한 파싱 실패**
- **Given**: 카메라 렌즈가 오염되었거나 역광 상태에서 촬영된 이미지가 주어진다
- **When**: `POST /api/v1/log-entries`로 이미지를 전송한다
- **Then**: Vision 파싱이 실패(confidence < 0.5)하여 LOG_ENTRY에 저장되지 않는다. `422 Unprocessable Entity` + `{ error: "VISION_PARSE_FAILED", message: "이미지 파싱에 실패했습니다. 재촬영을 요청합니다", retry_guidance: "렌즈를 닦고 역광을 피해 재촬영하세요" }` 반환. 파싱 실패 이벤트가 발행되어 E1-CMD-004에서 "재촬영 요청" 알림을 발송할 수 있다. **오 데이터 기록 0건 (오류 방어율 100%)**.

**Scenario 3: 이미지 파일 형식/크기 검증 실패**
- **Given**: 지원하지 않는 이미지 형식(BMP)이거나 15MB 초과 이미지가 주어진다
- **When**: `POST /api/v1/log-entries`로 전송한다
- **Then**: `400 Bad Request` + `{ error: "INVALID_IMAGE", message: "지원하지 않는 이미지 형식이거나 파일 크기(최대 10MB)를 초과합니다" }` 반환. LOG_ENTRY 생성 없음. 시스템 크래시 0건.

**Scenario 4: Gemini Vision API Rate Limit 도달**
- **Given**: Gemini API 호출이 15 RPM 한도에 도달한 상태이다
- **When**: 추가 Vision 파싱 요청이 인입된다
- **Then**: In-memory Queue에 대기열 추가. `202 Accepted` + `{ status: "QUEUED", estimated_wait_seconds }` 반환. 최대 대기 60초. 큐 드롭 0건.

**Scenario 5: Gemini Vision API 타임아웃**
- **Given**: Gemini API가 8초 이내에 응답하지 않는다 (REQ-NF-002: p95 ≤8,000ms)
- **When**: Vision 파싱 요청이 타임아웃된다
- **Then**: 1회 자동 재시도 → 재실패 시 `503 Service Unavailable` 반환. 감사 로그에 타임아웃 기록. 오 데이터 기록 0건.

## :gear: Technical & Non-Functional Constraints
- **성능**: Vision 전 과정(API 호출 + 파싱 + DB 저장) p95 ≤8,000ms (REQ-NF-002, Gemini Multimodal 응답 시간 반영)
- **Rate Limit**: Gemini Free Tier 15 RPM 엄수. STT(E1-CMD-001)와 Vision이 동일 큐 공유 (ASM-10)
- **Throttle 대기**: 최대 60초, 진행률 UI 필수 (REQ-NF-008)
- **보안**: RBAC OPERATOR 이상만 접근. Prisma 감사 로그 전수 기록 (REQ-NF-023)
- **AI 추상화**: Vercel AI SDK 인터페이스로 구현. `AI_PROVIDER=ollama` 전환 시에도 동일 인터페이스 유지 (ADR-9)
- **이미지 저장**: raw_data 필드에 이미지 메타데이터만 저장. 원본 이미지는 Supabase Storage(1GB 제한) 업로드 (REQ-NF-009)
- **서버 실행 시간**: Vercel Serverless ≤10초 제한 준수 (CON-07). 이미지 전처리를 최소화하여 시간 확보
- **데이터 무결성**: 파싱 실패 시 절대 저장하지 않음. 오 데이터 기록 0건 (REQ-FUNC-006)
