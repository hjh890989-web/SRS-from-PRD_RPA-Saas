## :white_check_mark: Task Breakdown (실행 계획)

### Phase A. Prisma Enum 및 Model 정의

- [ ] **A-1. `Role` Enum 정의**: `prisma/schema.prisma`에 사용자 역할 Enum 선언
  ```prisma
  enum Role {
    ADMIN      // 시스템 관리자 — 전체 시스템 설정, 사용자 관리, 모든 데이터 접근
    OPERATOR   // 현장 작업자/반장 — 로깅 입력, Approve/Reject, 롤백/수정
    AUDITOR    // 구매본부장/품질이사 — 감사 리포트 생성, XAI 대시보드 조회
    VIEWER     // 일반 조회 — 대시보드/리포트 조회 전용 (쓰기 불가)
    CISO       // 최고정보보안책임자 — RBAC 관리, 감사 로그, 네트워크 모니터링, 단독 거부권
  }
  ```

- [ ] **A-2. `User` Model 정의**: SRS §6.2.14의 8개 필드 + 보안용 `password_hash` 필드를 포함한 모델 선언
  ```prisma
  model User {
    id            String    @id @default(cuid())
    name          String    @db.VarChar(255)
    email         String    @unique @db.VarChar(255)
    password_hash String    // Bcrypt 해시 저장 전용. 평문 저장 절대 금지
    role          Role      @default(VIEWER)
    factory_id    String?   // FK → Factory (Factory 스키마 DB-003 완료 후 relation 추가)
    is_active     Boolean   @default(true)
    created_at    DateTime  @default(now())
    updated_at    DateTime  @updatedAt
    last_login_at DateTime?

    // Relations (후행 태스크에서 순차 추가)
    // factory     Factory?    @relation(fields: [factory_id], references: [id])  // DB-003 완료 후 활성화
    // approvals   Approval[]  // DB-008 완료 후 활성화
    // audit_reports AuditReport[] // DB-010 완료 후 활성화

    @@map("users")
  }
  ```

  > :warning: **설계 결정 사항**:
  > - **`id` 타입**: SRS는 UUID를 명시하나, Prisma + SQLite(dev) 호환성과 분산 ID 고려로 `cuid()` 채택 (ADR-10 범위). PostgreSQL(MVP)에서도 정상 동작
  > - **`password_hash` 추가**: SRS §6.2.14 원본에는 없으나, AUTH-001(NextAuth.js 자격 증명 프로바이더) 구현 시 필수. 평문 저장 절대 금지 (REQ-NF-022)
  > - **`updated_at` 추가**: SRS 원본에는 없으나, Prisma 표준 관례이며 감사 추적에 필수
  > - **`factory_id` nullable**: DB-003(FACTORY 엔터티)이 아직 미완료 상태이므로, relation 없이 `String?`으로 선언. DB-003 완료 후 FK relation 활성화

- **When**: `User` 모델 및 `Role` Enum을 `schema.prisma`에 선언하고 `npm run db:migrate:dev -- --name add_user_entity`를 실행함
- **Then**: `prisma/migrations/` 디렉토리에 마이그레이션 SQL 파일이 생성되고, `dev.db`에 `users` 테이블이 9개 컬럼(id, name, email, password_hash, role, factory_id, is_active, created_at, updated_at, last_login_at)으로 정상 생성된다. `npx prisma generate` 실행 후 Prisma Client에서 `User` 타입과 5가지 `Role` Enum이 import 가능하다.

### Scenario 2: 이메일 Unique 제약조건 검증

- **Given**: DB에 이메일이 `admin@factoryai.dev`인 User 레코드가 존재함
- **When**: 동일한 이메일(`admin@factoryai.dev`)을 가진 새로운 User 레코드를 `prisma.user.create()`로 INSERT 시도함
- **Then**: Prisma `P2002` 에러(Unique constraint violation)가 발생하며 레코드 생성이 차단된다. 기존 레코드는 영향 없이 보존된다.

### Scenario 3: 기본값(Default) 자동 적용 검증

- **Given**: `name`, `email`, `password_hash` 필드만 제공하고 `role`, `is_active`, `created_at`은 명시하지 않음
- **When**: `prisma.user.create({ data: { name: '테스트', email: 'test@test.com', password_hash: '$2b$...' } })`를 실행함
- **Then**: 생성된 레코드에서 `role`=`VIEWER`, `is_active`=`true`, `created_at`=현재 시각이 자동 적용된다.

### Scenario 4: Not Null 제약조건 위반 시 거부

- **Given**: `name` 또는 `email` 필드를 null로 설정한 User 생성 요청이 있음
- **When**: `prisma.user.create({ data: { name: null, email: null, ... } })`를 실행함
- **Then**: Prisma 유효성 검증 에러가 발생하며 레코드 생성이 거부된다. DB에 불완전한 데이터가 저장되지 않는다.

### Scenario 5: 5가지 Role Enum 전수 검증

- **Given**: `Role` Enum이 `ADMIN`, `OPERATOR`, `AUDITOR`, `VIEWER`, `CISO` 5가지 값을 가짐
- **When**: 각 역할로 1명씩 5명의 User를 생성하고, `prisma.user.findMany()`로 전체 조회함
- **Then**: 5건의 User 레코드가 조회되고, 각 레코드의 `role` 값이 정확히 해당 Enum 값과 일치한다. 미정의된 역할(예: `SUPERADMIN`)은 Prisma 타입 시스템에서 컴파일 에러로 차단된다.

## :memo: Implementation Notes (개발자 참고)

### SQLite Enum 호환성

Prisma는 SQLite에서 Enum을 네이티브로 지원하지 않고 **문자열(TEXT)**로 저장합니다. PostgreSQL에서는 네이티브 `ENUM` 타입으로 생성됩니다. Prisma가 양 환경 간 차이를 자동 처리하므로 개발자가 별도 대응할 필요는 없으나, **직접 SQL 쿼리 작성 시** 이 차이를 인지해야 합니다.

```
// SQLite: role 컬럼은 TEXT 타입으로 저장 ("ADMIN", "OPERATOR" 등 문자열)
// PostgreSQL: role 컬럼은 "Role" ENUM 타입으로 저장
```

### `factory_id` FK 관계 후속 처리 (DB-003 완료 후)

DB-003(FACTORY 엔터티) 완료 시, `User` 모델의 `factory_id` 필드에 `@relation`을 추가하는 후속 마이그레이션이 필요합니다:


### `password_hash` 보안 주의사항

- **절대 평문 저장 금지**: `password_hash` 필드에는 반드시 Bcrypt 등 단방향 해시만 저장
- **Seed 데이터**: 개발용 Seed에서는 placeholder 해시 사용. 실제 운영 시 AUTH-001에서 Bcrypt 모듈로 교체
- **로깅 마스킹**: API 요청/응답 로깅 시 `password_hash` 필드는 반드시 마스킹 처리 (REQ-NF-023 연계)
